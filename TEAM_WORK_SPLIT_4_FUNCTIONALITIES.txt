ERP INTEGRATION SUBSYSTEM
EQUAL WORK DISTRIBUTION FOR 4 PEOPLE

Purpose of this file:
- divide the entire Integration subsystem project among 4 people
- cover both local deployment and RDS deployment
- keep the work balanced in difficulty, implementation effort, testing effort, and presentation responsibility
- ensure every person gets a meaningful mix of design, implementation, validation, and documentation work

Project scope considered:
- studying subsystem data requirements
- reconciling all schemas into one shared database
- building and testing the local MySQL setup
- building the Java SDK and JAR
- implementing access control, logging, pooling, and exceptions
- deploying the database to Amazon RDS
- preparing team-facing configuration and usage documents
- supporting other subsystems in using the shared deployed database

Target work split:
- Person 1: about 25%
- Person 2: about 25%
- Person 3: about 25%
- Person 4: about 25%

==================================================
PERSON 1
SCHEMA ANALYSIS, DATA RECONCILIATION, AND SQL DESIGN
==================================================

Main role:
- own the database design side of the project

Detailed responsibility:
- This person studies the data requirements and SQL files of all subsystems and converts them into one final shared schema for the Integration subsystem.
- This includes comparing table names, columns, relationships, and data conflicts across teams.
- This person is also responsible for deciding where canonical tables are used and where compatibility views are needed.
- Their work directly affects both local and RDS deployments, because the entire project depends on the correctness of the final schema.

Tasks assigned:
- read all team SQL files in DbOfTeams
- compare subsystem table names, columns, keys, and references
- identify conflicting or duplicate structures across teams
- prepare the unified schema design
- maintain sql/01-schema.sql
- maintain sql/02-verification.sql
- add compatibility views for mismatched subsystem names
- maintain permission_matrix and data_ownership SQL seed data
- validate that table structure supports all required CRUD operations
- check alignment between schema and subsystem data requirements

Main deliverables:
- final schema design
- compatibility table/view mapping
- schema verification SQL
- subsystem alignment notes

Why this is a fair share:
- it is high-responsibility design work
- it requires understanding every subsystem, not just one
- it carries both technical depth and cross-team dependency

Estimated share:
- 25%

==================================================
PERSON 2
LOCAL DATABASE SETUP, SCRIPTING, AND VALIDATION
==================================================

Main role:
- own the local execution environment and database validation

Detailed responsibility:
- This person makes sure the full schema works properly on a local MySQL server and that the project can be demonstrated reliably.
- They are responsible for Docker setup, local port handling, schema loading, reseeding, troubleshooting SQL execution issues, and validating the seeded database state.
- They also help prove that the integration subsystem works before anything is moved to RDS.

Tasks assigned:
- prepare and maintain docker-compose.yml
- configure local MySQL container setup
- handle local DB creation and schema reload process
- maintain scripts/setup-local-db.ps1
- maintain scripts/reload-schema.ps1
- run sql/02-verification.sql and check outputs
- troubleshoot local DB errors such as missing columns, view issues, or load failures
- validate audit_logs, permissions, and seeded metadata locally
- prepare demo-ready local execution steps
- capture proof that the local environment works correctly

Main deliverables:
- working local MySQL setup
- local deployment scripts
- verification outputs
- demo-ready local setup instructions

Why this is a fair share:
- setup and debugging take significant effort
- local validation is required before cloud deployment
- this part includes both implementation support and repeated testing work

Estimated share:
- 25%

==================================================
PERSON 3
JAVA SDK, ACCESS CONTROL, AND JAR DELIVERY
==================================================

Main role:
- own the Java implementation that other subsystems will actually use

Detailed responsibility:
- This person develops the Java integration layer that sits between the teams and the database.
- Their work includes subsystem facades, CRUD support, access control, exceptions, connection pooling, join handling in code, and logging.
- They are also responsible for building the final JAR and verifying that the subsystem classes work properly against the database.

Tasks assigned:
- implement and maintain subsystem facade classes
- implement factory pattern logic
- implement adapter-related integration pieces
- implement CRUD methods for subsystem-facing operations
- handle joins and internal database operations inside code
- implement access-control checks and exceptions
- integrate HikariCP connection pooling
- ensure audit logging is triggered through Java operations
- support backup and restore integration from Java side
- maintain scripts/run-demo-checks.ps1
- maintain pom.xml build setup
- generate the final JAR for distribution

Main deliverables:
- Java SDK source code
- buildable Maven project
- target/erp-subsystem-sdk-1.0.0.jar
- runnable demo checks

Why this is a fair share:
- this is the main implementation-heavy part of the project
- it includes OOAD patterns, runtime logic, and external team usage support
- the JAR is one of the final submission outputs

Estimated share:
- 25%

==================================================
PERSON 4
RDS DEPLOYMENT, TEAM INTEGRATION, AND DOCUMENTATION
==================================================

Main role:
- own the cloud deployment and the handoff to all other teams

Detailed responsibility:
- This person takes the completed subsystem from local execution to live shared usage through Amazon RDS.
- They handle loading the final schema into RDS, validating remote connectivity, preparing the config file that other teams will use, and writing the instructions for using the shared JAR.
- They also support integration with other subsystem teams and document how to test that no team is accessing the database directly.

Tasks assigned:
- deploy the final schema to Amazon RDS
- verify connectivity from local machine to RDS
- prepare and maintain the shared RDS configuration file
- maintain README_FOR_OTHER_TEAMS_RDS.md
- maintain docs/DEPLOYED_DATABASE_USAGE.md
- prepare subsystem-specific integration guides when needed
- explain how teams should use the JAR instead of direct DB queries
- coordinate with teams during integration testing
- prepare final usage instructions for faculty/demo
- document remaining tasks, known limitations, and support steps

Main deliverables:
- working RDS deployment
- shared config file for teams
- RDS usage README
- integration handoff documents
- presentation and demo support documents

Why this is a fair share:
- this part involves deployment, coordination, and production-style usage
- it is essential for making the subsystem live and reusable
- it includes both technical and cross-team responsibility

Estimated share:
- 25%

==================================================
HOW THE TOTAL PROJECT LOAD IS BALANCED
==================================================

This split is balanced because each person gets:
- one major ownership area
- implementation work
- testing or validation work
- documentation or explanation work
- visible deliverables for presentation

Balance summary:
- Person 1 carries the core schema and reconciliation design load
- Person 2 carries the local execution and troubleshooting load
- Person 3 carries the Java implementation and JAR delivery load
- Person 4 carries the cloud deployment and team integration load

No person is assigned only support work.
No person is assigned only documentation work.
Every person has a clear, important, and demo-worthy contribution.
